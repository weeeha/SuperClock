# SuperClock documentation site

The public system map and per-app documentation, deployed at
**https://superclock-map.vercel.app**.

It is a static site generated by a small Node script — no framework, no
dependencies, no build tooling beyond Node itself.

## Layout

| Path | What it is |
|---|---|
| `data.js` | Per-app content: description, sitemap tree, key flows, remaining tasks, files. **Edit this to change an app's page.** |
| `index.src.html` | The system-map survey — concept map, sitemap, API surface, faces, fleet, gap evaluation. Hand-written. |
| `build.js` | Generator. Extracts the stylesheet, emits `/docs`, `/apps`, and one page per app. |
| `capture.cjs` | Playwright script that screenshots all 12 apps from the dev server. |
| `shots/*.png` | Committed 1080×1080 screenshots. Regenerate with `capture.cjs`. |
| `dist/` | Generated output. Gitignored — never edit by hand. |

## Build

```bash
node build.js          # -> dist/
```

Re-running is safe; `dist/` is rebuilt from scratch each time.

Output routes: `/` redirects to `/docs`; `/apps` is the catalogue; `/apps/<id>`
is one page per app. Clean URLs and the redirect are configured in the
generated `dist/vercel.json`.

## Refreshing the screenshots

The capture script drives the real kiosk via its navigation store, so the
screenshots are genuine renders rather than mockups — including the offline
states for apps whose data source isn't configured.

```bash
npm run dev                    # in the repo root, serves on :5180
node capture.cjs               # needs `playwright` resolvable
```

`capture.cjs` stays CommonJS on purpose: the repo is `"type": "module"`, and
Playwright is usually not a dependency here, so it is loaded with `require`
from wherever it happens to be installed. If `playwright` is not resolvable,
either add it (`npm i -D playwright`) or point `NODE_PATH` at an existing
install before running.

It writes one PNG per app into `shots/`, plus `app-grid.png`. Apps needing a
network source (weather, calendar, github, claude-usage) will capture their
offline state unless those are configured — which is usually what you want the
documentation to show.

## Deploy

`dist/` is a self-contained static folder. Deploy it directly:

```bash
node build.js
cd dist && vercel deploy --prod --scope nick-vyhouskis-projects
```

There is no Vercel build step — the project serves `dist/` as-is.
