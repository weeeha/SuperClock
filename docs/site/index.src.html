SuperClock: what's built, what's wired, what's still missing

A read of the repository as it stands on main — twelve kiosk apps, nine watch faces, a fleet admin panel, and an mmWave radar that the software is ready for and the firmware isn't. This maps the concepts, the navigable surfaces, and the ten gaps that actually matter.

Concept map

Two relationships carry the whole system. The first is what a screen is — a chain from the fleet file down to a single complication. The second is how a change travels — from a phone in the hallway to a Pi on a shelf.

Domain model. Note the two dashed edges — they're where the design outruns the code. The schema registry validates every instance's config, but the Complication node at the far right is almost entirely declaration: six types are registered so the admin's slot picker has something to show, one draws live data, and eight of the nine faces declare no slots at all.

Config pipeline. Every Pi is standalone-functional by design: the localStorage last-good cache means a kiosk that boots while the admin host is down still shows its configured screens. Failed pushes retry on a 60-second loop rather than failing the admin write.

Sitemap

Two SPAs ship from one Vite build and one Express server. The kiosk has no router at all — navigation is a Zustand store plus a single root gesture handler. The admin is conventional react-router.

Gesture contract

One @use-gesture handler at the root, using pointer events with pointer capture — so child components cannot intercept it by stopping touch-event propagation. Vertical swipe routes to the active app's registered callback when it has one, and opens the grid when it doesn't. There is no metadata flag for this; apps call setVerticalSwipeCallback while active. A long-press replacement for pinch and 3-finger tap sits on an unmerged branch.

API surface

Express 5. Every third-party secret is server-side — the browser never sees a token. Unmatched /api/* returns JSON 404 before the SPA fallbacks, so a typo never silently renders the kiosk.

The twelve apps

Each is a module under src/apps/ that registers itself and lazy-loads. Adding one means touching three lists — the side-import, the capability array, and the schema registry — and a coherence test fails until they agree.

Clock clock

Shipped

The most developed surface. A face-cycling shell hosting nine swappable faces; vertical swipe moves between them. Accepts a faceId plus merged face options from device config, so the admin can pin a device to one face.

utilityconfig: face-driven

Weather weather

Shipped

Current conditions and forecast from Open-Meteo — no API key needed. Shows an explicit offline tell rather than stale numbers. An hourly radial view is in design, not built.

utilityapp.weatherclient fetch

Calendar calendar

Expanding

Today's date and events from any iCal feed, parsed server-side. PR #28 adds Month, Week and Details views laid out for the round display, plus the repo's first app-level test file.

utilityapp.calendar/api/calendar

Fitness fitness

Thin

Today it's an exercise counter with a progress ring. The architecture doc names the real Fitness — groups, sets, active circuit — as the next build, because it's the one app that would force five shared primitives into existence at once.

productivityapp.fitness

GitHub github

Shipped

Contribution heatmap arranged as a radial watch face. Recently hardened: serves a last-good cache when offline, shows an honest empty state, and no longer falls back to mock data.

productivityapp.githubserver proxy

Habits habits

Shipped

Daily habit tracking with streaks; vertical swipe switches day and month views — one of only two apps using the tier-2 pattern. A radial-daily / concentric-monthly rewrite is stranded on an unmerged branch.

productivityapp.habits

Timer time-tracking

Radar-aware

Pomodoro focus timer, and the only app that consumes occupancy: it auto-pauses after 45 seconds of radar absence and resumes when you come back. Reads today's at-desk total and a 7-day history.

productivityapp.time-tracking/api/occupancy

Claude Usage claude-usage

Needs daemon

Session and weekly rate-limit utilization with a mood-aware Clawd character. Depends on a Mac LaunchAgent holding a valid subscription token — when the numbers vanish, that daemon is down, not the app.

productivityapp.claude-usageMac :47823

Breathing breathing

Blocked

Live respiration rate from the A121 mmWave radar, leasing breathing mode for 90 seconds at a time. The software path works end to end against the mock driver — but on real hardware the mode switch sends nothing. See gap 4.

utilityapp.breathing/api/radar

Photos photo-frame

Shipped

Slideshow cycling local photos every eight seconds. The photo directory is gitignored and copied into the build; albums are selectable per device.

ambientapp.photo-frame

Fireplace fireplace

Shipped

Ambient fireplace animation. One of the three ambient apps that the architecture doc suspects should collapse into a single scene-picker app.

ambientapp.fireplace

Quote quote

Shipped

Quote of the day, optionally sourced from a remote JSON feed. The simplest app in the repo, and the closest thing to a template for a new one.

Complications

Six are registered so the admin's slot picker has a catalog: Date, Temperature, Habit Streak, Day Progress, Fitness Ring, Next Event. Only Date and Temperature have renderers, and Temperature renders an em-dash because nothing passes it a value yet. Combined with the slot column above, that means no complication currently appears on any screen. This is gap 1.

The fleet

Four Pis. Three run Chromium in kiosk mode against the React build; the fourth runs a separate native binary and is read-only from the admin's point of view.

The Chromium kiosk is deliberately not a systemd service — only the Express server is. The browser is launched from the labwc autostart script, which waits for /api/health first. The server ships as a single esbuild bundle, so deployment has no list of source directories to keep in sync.

Gap evaluation

Ranked by how much each one blocks. The pattern across the top three is the same: the infrastructure for a capability was built before anything consumed it, so the admin now offers controls that don't reach the screen.

01

The complication system is a facade

Promises what it can't deliver

Six complications are registered, one draws live data, and eight of nine faces declare slots: []. The admin's slot grid lets you assign a complication to a screen that will never render it — the worst kind of gap, because it looks finished.

FixEither wire two or three real slots into Complications Dark, which already declares four demo slots and a config schema, or gate unimplemented complications out of the picker so the admin stops promising them.

02

The primitive kit is zero of seven built

Blocks scale

RingFace, RadialViz, ScenePicker, IntervalTimer, RoundList, WeekRing, AmbientMedia — all named as the foundation for the roughly fifteen apps in design. None exist; src/core/widgets/ doesn't exist. Nor do the five shell hooks, so every app re-rolls its own fetch, interval and localStorage handling, and TanStack Query is a dependency the kiosk never adopted.

FixThe doc already answers this: build Fitness for real and extract on the third use. Fitness is the right forcing function precisely because it needs five of the seven at once.

03

Navigation tiers 2 and 3 live in app code, not the shell

Blocks scale

Tier 2 is a callback registration each app copies by hand — two apps do it today, eight-plus planned apps want it. Tier 3, the drill-down back stack, has no implementation at all, and the real Fitness app needs three levels.

FixLift both into the shell alongside the Fitness build, so the API is validated by a real call site instead of designed in the abstract.

04

Radar mode switching is a no-op on real hardware

Blocks Breathing

The serial driver's setMode logs a line and sends nothing — the UART register protocol was never calibrated because the module wasn't in hand. Presence and breathing are mutually exclusive in the flashed firmware, so the Breathing app cannot work on a real sensor today, only against the mock.

FixPR #29 is exactly this — a 259-line Python exploration-server sidecar that moves the DSP host-side. It's the one open PR with a physical dependency, so it needs the sensor plugged into a Pi to validate.

05

Faces are curated React while the stated direction is JSON config

Blocks horizons 2 & 3

Every new face is a pull request and a deploy. The foundation's own lean is option B1 — faces as pure config on top of engines — and the infrastructure is already there: a face registry, per-instance config, and a zod-to-form renderer. What's missing is a face spec that something consumes.

FixThis unblocks the community face library, the visual face builder, and gap 6 simultaneously. It's the highest-leverage mid-term move.

06

React and LVGL parity is maintained by hand

Silent drift risk

Only Minimalismo exists on both sides. Any change to a shared face's geometry, palette or night behaviour needs a mirrored edit in C in the same PR — enforced by review discipline alone, with nothing failing if it's forgotten.

FixDownstream of gap 5. A shared JSON face spec is the stated long-term answer; until then this stays a review checklist item.

07

Four test files, none covering an app or a route

Least-verified code is the newest

The four that exist pin the right things — fleet store atomicity, the navigation invariant that would kill every gesture, registry coherence, time windows. But there are no API-route tests and no radar protocol tests, and the radar parser is both the newest and the least verifiable code in the repo.

FixPR #28 adds the first app-level test. Radar protocol tests should land with the sidecar, using captured serial lines as fixtures.

08

The README is stale and contradicts the security rule

Blocks open-sourcing

It documents VITE_GITHUB_TOKEN — a browser-inlined secret — when the token moved server-side to a proxy route. It says eleven apps when twelve are registered, and never mentions the admin panel, the fleet pipeline, or the radar. The project docs are accurate; the README is the file an outside visitor reads first, and horizon 2 is literally "open-source the repo".

FixA rewrite is an hour of work and a prerequisite for the stated mid-term goal.

09

Sixteen unmerged branches, including a PR that predates the radar

Hygiene

Two salvage branches carry an in-kiosk Settings panel that doesn't exist on main; others hold a long-press gesture replacement and the Habits rewrite. More urgently, PR #28 branched before the radar merge — its diff against main shows the radar files as deletions, so it will read as a revert unless it's rebased first.

FixRebase PR #28 before review. Then decide the salvage branches in or out — they're currently neither.

10

No device-to-device channel

Horizon 3, unstarted

"Push a scene to the other room" has no transport. Admin-to-device is one-way with a retry loop, and the fleet store already knows every device — so the missing piece is genuinely small, it just hasn't been started.

FixNot urgent. Worth noting that nothing in the current design forecloses it, which was the point.

What I'd do next

In order. The first three are small and clear the deck; the fourth is the one that actually changes the trajectory.

  1. Rebase PR #28, then land it

    It's eighteen commits of real work sitting behind a stale base. Rebasing first stops the radar files reading as a revert.

  2. Validate PR #29 against the physical sensor

    Everything downstream of the protocol parser already works. This is the only item on the list gated on hardware rather than time, so it's the one worth starting early.

  3. Make the complication system honest

    Either wire two real slots into Complications Dark or hide the unimplemented ones. Right now the admin can configure something that cannot appear.

  4. Build the real Fitness app and extract as you go

    The single highest-leverage move in the repo. It forces RoundList, IntervalTimer, ScenePicker, WeekRing and the tier-3 back stack into existence against a real call site. If Fitness lands small because the kit is right, the model is proven; if the primitives don't fall out naturally, that's worth knowing before fifteen more apps are built on the assumption.

  5. Then: face spec as data, and a README rewrite

    These two together are what "open-source the repo" actually requires. Neither is urgent this week; both are prerequisites for the horizon-2 goal.
